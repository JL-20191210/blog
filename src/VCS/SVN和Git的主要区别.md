# SVN和 Git 的主要区别

> Git clone的是整个项目，本地保存了所有的分支和历史记录
>
> svn本地只保存了一个分支

## 1. **版本控制模型**：
   - **SVN**：集中式版本控制（Centralized Version Control System, CVCS）。
     - 所有版本历史和代码都存储在中央服务器中，用户必须从服务器中“检出”（checkout）工作副本。
     - 修改的文件会提交到中央仓库，其他用户需要更新（update）工作副本才能看到这些更改。
   - **Git**：分布式版本控制（Distributed Version Control System, DVCS）。
     - 每个用户都拥有整个仓库的完整副本，包括历史记录、分支等，操作完全可以在本地完成。
     - 本地更改可以先在本地提交，然后再推送（push）到远程仓库。

## 2. **工作方式**：
   - **SVN**： 
     - 本地只保留当前版本的工作副本。
     - 所有操作必须与中央服务器交互，必须在线进行提交和更新。
   - **Git**： 
     - 本地存储整个仓库，包括所有分支和历史记录。
     - 本地可以进行各种操作（如提交、切换分支等），不需要与远程仓库交互，除非需要推送或拉取更改。

## 3. **分支管理**：
   - **SVN**： 
     - 分支管理相对复杂，分支是通过复制（`svn copy`）实现的，需要手动创建分支，且切换分支需要执行 `svn switch`。
     - 分支合并有时较为繁琐，尤其在多人协作时容易出现冲突。
   - **Git**： 
     - 分支非常轻量且操作简便，可以在本地随时创建和切换分支（`git branch` 和 `git checkout`）。
     - 分支管理更为灵活且高效，分支合并操作通常比 SVN 更加简单。

## 4. **网络依赖性**：
   - **SVN**：
     - 必须依赖网络与中央服务器进行操作（例如提交、更新、检出）。
     - 在没有网络连接时，无法进行版本控制操作。
   - **Git**：
     - 操作大部分都可以在本地进行，只有推送（push）和拉取（pull）时才需要与远程仓库交互。
     - 即使没有网络连接，开发者依然可以本地提交和查看历史。

## 5. **历史记录**：
   - **SVN**：
     - 历史记录保存在中央仓库中，只能查看当前分支的历史记录。
     - 需要使用 `svn log` 来查看提交历史。
   - **Git**：
     - 每个本地仓库都包含完整的提交历史，可以随时查看任何分支和任何版本的历史。
     - 使用 `git log` 可以查看历史记录，且能查看整个仓库的所有分支历史。

## 6. **性能**：
   - **SVN**：
     - 因为操作都依赖于中央服务器，网络状态和服务器性能会影响操作速度。
     - 在进行大量的提交和更新时，效率相对较低。
   - **Git**：
     - 本地操作速度非常快，因为大多数操作都是本地完成的，不依赖网络。
     - 操作如提交、切换分支、查看历史等通常比 SVN 快得多。

## 7. **合并与冲突管理**：
   - **SVN**：
     - 分支合并比较复杂，尤其是在大型项目中，容易发生冲突。
     - 需要手动解决冲突，SVN 的冲突管理不如 Git 灵活。
   - **Git**：
     - 分支合并较为容易，冲突处理也更加高效。
     - Git 会自动标记冲突并允许开发者选择如何解决。

## 8. **适用场景**：
   - **SVN**：
     - 更适合于大型团队中管理少数几个重要的分支和代码版本，特别是在需要严格控制访问权限的情况下。
     - 适合那些不太需要频繁分支和合并的项目。
   - **Git**：
     - 更适合需要频繁创建分支和进行版本切换的小团队或开源项目。
     - 适用于快速迭代和分布式开发，开发者可以在本地独立进行版本控制操作。

## 9. **工具支持**：
   - **SVN**： 
     - 支持的工具较少，主要有 TortoiseSVN（Windows）和其他命令行工具。
   - **Git**：
     - 拥有非常广泛的工具和集成支持，如 GitHub、GitLab、Bitbucket，以及许多 IDE（如 VS Code、IntelliJ IDEA）都提供 Git 集成。

## 10. **学习曲线**：
   - **SVN**：
     - 相对简单，命令较少，适合传统集中式管理的项目。
     - 学习曲线较平缓，但不支持分布式操作的灵活性。
   - **Git**：
     - 学习曲线较陡，特别是在分支管理和合并方面。
     - 但一旦掌握，Git 提供的灵活性和强大功能能显著提高工作效率。
